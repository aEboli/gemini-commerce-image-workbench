# 网页生图项目分析-2026-03-12

对应对象：基于 Gemini 的电商 AI 出图工作台

## 1. 当前阶段判断

项目当前已经明显不再是“单点出图页面”的原型，而是一个围绕电商素材生产流程搭建的桌面式工作台。结合 PRD、版本说明、页面结构与后端链路来看，它已经完成了从“能生成”到“能交付、能协作、能回看”的跃迁。

当前更准确的阶段判断是：

- 产品阶段：`v0.3.1` 可交付版本，适合内部团队或小团队实际使用。
- 研发阶段：核心能力基本成型，当前主要工作重心已经转向稳定性、体验压缩、结果管理和协同闭环。
- 架构阶段：单机/局域网优先的本地化架构成立，但若继续扩大使用范围，任务调度、可观测性和依赖稳定性会成为下一阶段的主要瓶颈。

## 2. 已形成的核心能力

### 2.1 创作能力已经比较完整

- 支持标准出图、提示词模式、参考图复刻、套图模式、亚马逊 A+ 图模式。
- 支持国家、语言、平台、比例、分辨率、数量、品牌、品类、卖点、材质、尺寸等结构化输入。
- 支持模板自动匹配与手动覆盖，说明生成逻辑已经具备一定“运营规则化”基础。

### 2.2 结果管理链路已经闭环

- 有历史记录页、任务详情页、结果筛选、入选/通过/淘汰、通过项打包下载。
- 任务详情页已经不是简单结果列表，而是带有输入快照、对比、复用 prompt、飞书重同步的结果工作台。
- 历史记录页已经具备缩略图预览、放大查看、统计与耗时展示，说明项目开始重视回看效率而不只是生成成功率。

### 2.3 协作与交付能力已经落地

- 飞书多维表格同步已接入，且不仅是“上传一次”，还支持字段映射、重同步和大图上传重试。
- 已具备 Windows 安装器和安全发布思路，说明项目目标不是纯研发演示，而是实际分发给非开发同事使用。

### 2.4 最近迭代方向很清晰

最近的迭代集中在四件事：

- 参考图复刻稳定性修复
- Gemini 分辨率档位与实际返回尺寸校验
- 创作台首屏压缩与参数交互优化
- 历史记录与详情页的“工作台化”

这说明你现在已经从“补功能”切换到了“补真实使用体验”阶段，这个方向是对的。

## 3. 当前项目的强项

### 3.1 产品思路不是通用生图，而是电商工作流导向

项目不是一个泛用图片生成器，而是按电商素材类型、市场、平台、协作方式组织能力。这一点会让后续迭代更容易持续拉开和通用 AI 图像工具的差异。

### 3.2 提示词治理已经有明显产品化意识

- 空字段不再发送给模型
- 中文统一约束简体中文
- 材质、尺寸、卖点按图位隔离进入不同提示词
- 参考图复刻支持结构分析与文案槽位

这些都说明你已经开始把“提示词工程”沉淀成业务规则，而不只是随手拼 prompt。

### 3.3 前端体验在朝真正可用的操作台演进

从总览页、创作台、历史记录到详情页，界面都明显在往“高频使用的操作台”方向收敛，而不是一次性展示页。这一点对内部长期使用很关键。

## 4. 当前最值得关注的风险

### 4.1 运行时稳定性仍然偏单机原型

- 任务队列仍是进程内内存队列。
- 应用重启后，队列上下文不会保留。
- 这对单人本地使用问题不大，但对安装器分发、多任务、长时间运行会有隐患。

### 4.2 数据层依赖 `node:sqlite` 实验能力

当前能跑、也已经通过构建，但 `next build` 仍会输出 `node:sqlite` experimental warning。短期可接受，长期如果继续作为正式桌面工具分发，最好逐步降低对实验能力的依赖。

### 4.3 发布信息存在版本不一致

`package.json` 已是 `v0.3.1`，版本说明也写的是 `v0.3.1`，但 `README.md` 仍在展示 `v0.3.0` 和旧安装器文件名。这会直接影响外部认知和对外交付的可信度。

### 4.4 自动化验证还偏弱

当前已做 `typecheck` 和 `build`，但仓库里暂未看到围绕以下关键链路的自动化测试：

- 任务创建校验
- 图位模板路由规则
- 尺寸档位判定与重试
- 飞书字段映射与同步容错
- 历史记录统计聚合

这会让后续每次迭代都更依赖人工回归。

### 4.5 文档沉淀存在编码/一致性风险

`doc/进展记录.md` 在终端读取时有乱码表现，虽然文件本体仍可继续维护，但后续最好统一文档编码与编辑方式，避免积累更多历史问题。

## 5. 优化建议

建议按优先级拆成三层来做。

### 优先级 A：先补稳定性

1. 把任务队列从“进程内内存队列”升级为“数据库可恢复队列”。
   目标不是上复杂中间件，而是至少做到应用重启后可恢复 `queued / processing` 状态，并能继续消费未完成任务。

2. 给关键生成链路补可观测性。
   建议最少补三类日志：任务级、生成项级、外部调用级。重点记录模型、尺寸档位、重试次数、失败阶段、飞书上传阶段。

3. 处理 Windows 下 `next build` 的 standalone copyfile 警告。
   这不一定马上影响使用，但它已经影响“安装器分发”的稳定预期，值得单独做一次构建链路清理。

### 优先级 B：再补效率与复用

1. 增加“历史任务二次生成”能力。
   现在已经能复用 prompt，如果再补“以历史任务为模板重新生成”，会很适合运营高频迭代。

2. 给模板和品牌库加命中解释。
   不只是显示“命中了什么模板”，而是说明“为什么命中”，这样更方便运营同学调规则。

3. 把详情页的优质结果沉淀为“可复用方案”。
   比如允许将某次成功结果直接保存为模板参考、品牌提示、参考图策略或 A+ 模块方案。

### 优先级 C：最后补增长与协同

1. 增加任务与结果层的运营指标。
   例如成功率、平均耗时、各图位失败率、各分辨率失败率、不同模式使用占比。

2. 增加“批量补同步飞书”和“仅补传失败项”。
   目前已支持单任务重同步，下一步很自然就是批量运营修复能力。

3. 如果后续继续做跨境运营场景，可以考虑把市场研究或竞品分析结果接到模板推荐前。
   这一步不必现在做，但会是从“出图工具”升级成“运营决策辅助工具”的关键方向。

## 6. 我对当前进展的结论

如果用一句话概括：

这已经是一个“围绕电商素材生产流程设计的本地化 AI 出图工作台”，而不是一个普通网页生图 Demo。

你现在最需要的不是再拼更多模式，而是把已经很强的能力做得更稳、更可复用、更可回溯。只要把任务恢复、自动化验证、发布一致性和协同修复这几件事补上，整个项目的完成度会再上一个台阶。

## 7. 建议下一步

如果你想按投入产出比来排，我建议下一步按这个顺序：

1. 统一版本与文档信息，先把对外交付口径收拢。
2. 单独处理队列持久化与失败恢复。
3. 给生成链路补一组最小自动化测试。
4. 再做历史任务复用与模板命中解释。
